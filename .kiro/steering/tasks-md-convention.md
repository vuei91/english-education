# tasks.md 규약

> `.kiro/specs/**/tasks.md` 가 컨텍스트에 들어오면 자동 포함.
> 태스크 상태 마커와 실행 단위의 의미를 고정한다.

## 체크박스 마커

| 마커 | 의미 | 오케스트레이터 동작 |
|------|------|--------------------|
| `- [ ]` | 미착수 | Run All Tasks 대상 |
| `- [~]` | 큐잉됨 | 실행 순서 확정, 아직 시작 전 |
| `- [-]` | 진행 중 | 서브에이전트 실행 중 |
| `- [x]` | 완료 | 건드리지 않음 |

별표 `*` 또는 `\*` 가 붙은 태스크는 **선택(optional)** 이며 Run All Tasks 자동 실행 대상이 아니다.

## 트리 구조

```
- [ ] N. 상위 태스크
  - [ ] N.M 서브 태스크 (Req X.Y, ...)
  - [ ] N.M 서브 태스크 (Req X.Y)
```

- 서브 태스크가 있는 경우 **리프(서브)부터** 완료한다. 모든 서브가 `[x]` 이면 상위를 `[x]` 로 올린다.
- 각 태스크는 1~3일 내 완료 가능한 단위. 그 이상이면 쪼개는 것을 제안한다.
- 요구사항 역추적은 괄호 안에 `(Req N.M, N.M)` 형태로 명시한다.

## 주석(quote) 블록

태스크 아래의 `>` 인용 블록은 **실행 메모**다:

- 왜 이 태스크를 특정 방식으로 축소/연기했는지
- 어떤 후속 태스크에 의존성을 남겼는지
- 임시 구현이 정식 구현으로 교체되어야 한다면 어느 태스크에서 하는지

실행자는 이 메모를 읽고, 판단에 영향이 있으면 그대로 따른다. 메모와 현재 요구가 충돌하면 오케스트레이터에 보고한다.

## 진행 중(`[-]`) 마커의 해석

`[-]` 는 두 가지 용법이 섞여 쓰인다:

1. **오케스트레이터가 실행 중** — 현재 세션에서 서브에이전트에 위임해 작업 중
2. **사람이 남긴 "부분 완료"** — 스펙 분리·기반 부재 등으로 이 스펙 내에서는 의도적으로 미완

둘을 구분하기 위해 `[-]` 태스크는 **직전 세션의 완료 보고** 또는 **quote 블록의 메모** 를 함께 읽어 판단한다. 메모가 "별도 스펙으로 분리", "Task X 에서 함께" 같은 지연 사유를 명시하고 있으면 Run All Tasks 자동 실행에서 제외한다.

## Run All Tasks 실행 전 체크리스트

1. `[ ]` + 비optional 태스크를 전부 수집한다
2. `[-]` 태스크 중 quote 메모가 "연기" 의미가 **아닌** 항목(즉 단순 진행중 미완)을 수집한다
3. 의존 순서에 따라 큐잉(`[~]`)
4. 각각 서브에이전트에 위임하며 상태를 `in_progress → completed` 로 전이
5. 실패 시 즉시 중단하고 사용자에게 보고

## 완료 판정 기준

태스크 하나가 `[x]` 로 전환되려면:

- 요구사항 ID 에 대한 구현 근거가 코드 또는 테스트에 존재
- 타입체크 / 린트가 실패하지 않음
- 관련 유닛 테스트 (있으면) 통과
- Non-Goals 리스트에 저촉되지 않음

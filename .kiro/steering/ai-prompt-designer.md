# AI Prompt Designer — LLM 용도별 프롬프트 설계 프로토콜

> `#ai-prompt-designer` 로 명시적으로 호출해서 로드한다.
> Groq Llama 3.3 70B / 3.1 8B 를 어느 용도에 쓸지, 프롬프트를 어떻게 구성하고 어떻게 검증할지 정의한다.

## 핵심 역할

**LLM 출력이 앱의 UX·학습 품질을 직접 결정하는 지점**(독해 퀴즈 채점, Vocab Helper, 청킹, 회화)의 프롬프트를 설계·검증·버전 관리한다.

## 작업 원칙

1. **모델은 용도에 맞게 배정한다** — 단순 반복은 8B, 품질이 UX를 가르는 지점은 70B
2. **프롬프트는 코드처럼 관리** — 버전, 변경 이력, 테스트 케이스 필수
3. **출력은 스키마로 제약** — 자유 서술 대신 JSON 스키마 또는 고정 구조로 받는다
4. **샘플 기반 검증** — 프롬프트를 배포하기 전 최소 20개 샘플에 대해 품질 평가
5. **비용은 항상 확인** — 배치 처리·캐싱·모델 다운사이징을 먼저 검토

## 용도별 모델 배정 (tech-stack.md와 동일)

| 용도 | 모델 | 형태 | 근거 |
|------|------|------|------|
| 주관식 독해 퀴즈 채점 | Llama 3.3 70B | 런타임 호출 | 의미 기반 피드백 품질이 학습 경험을 결정 |
| AI 회화 (Future) | Llama 3.3 70B | 런타임 호출 | 대화 자연스러움 |
| 독해 퀴즈 생성 | Llama 3.1 8B | 배치 사전 생성 | 다량 생성, 큐레이션 전제 |
| 긴 문장 자동 청킹 | Llama 3.1 8B | 배치 사전 생성 | 구조 분해, 반복 패턴 |
| Vocab Helper 생성 | Llama 3.1 8B | 배치 사전 생성 | 대량 어휘 처리 |

## 프롬프트 템플릿

### 공통 규격

```
[SYSTEM]
당신은 <역할>. 반드시 한국어로 답변한다. 출력은 아래 JSON 스키마만 따른다.

[출력 스키마]
{ ... }

[제약]
- 금지 사항 N가지
- 가독성 기준
- 저작권 주의

[USER]
<입력 데이터>
```

### 독해 퀴즈 채점 (런타임)

**입력**: 원문, 문제, 모범 답안, 사용자 답
**출력 스키마**:
```json
{
  "strengths": ["잘 짚은 점 1~3개"],
  "improvements": ["보완하면 좋을 점 1~3개"],
  "model_answer": "2~3문장 모범 답안",
  "accuracy_hint": "low | medium | high"
}
```
**원칙**:
- 점수·등급을 보여주지 않는다. `accuracy_hint`는 내부 SRS용
- 강한 부정 표현("틀렸다", "실패") 금지, 격려 톤 유지
- 사용자 답이 극도로 짧아도 3단 피드백 구조 유지

### 긴 문장 청킹 (배치)

**입력**: 영어 문장
**출력 스키마**:
```json
{
  "chunks": [
    { "text": "...", "depth": 0, "role": "subject | relative | prep_phrase | ..." }
  ]
}
```
**원칙**:
- depth는 종속절 깊이. 시각화에 사용
- role은 렌더링 힌트. 값 목록은 고정 enum으로 관리

### Vocab Helper 생성 (배치)

**입력**: 단어, 품사, 예문
**출력 스키마**:
```json
{
  "etymology": {
    "parts": [{ "text": "in-", "meaning": "안으로" }],
    "gloss": "안을 들여다보다",
    "related_words": ["respect", "suspect"]
  },
  "mnemonic": {
    "korean_phrase": "...",
    "story": "한 문장 설명"
  },
  "pronunciation_ipa": "..."
}
```
**원칙**:
- etymology는 Etymonline 교차 확인 필요 표시
- mnemonic은 성인 관점에서 무해해야 함 (이후 큐레이션)

## 검증 프로토콜

새 프롬프트 또는 프롬프트 변경 시 다음을 거친다:

1. **스키마 검증** — 20개 샘플 입력에 대해 출력이 100% 스키마 일치하는가
2. **품질 평가** — 동일 샘플을 `general-task-execution` 에 "품질 평가자" 프롬프트로 재주입해 팬아웃/팬인 평가
3. **회귀 샘플 세트 유지** — 이전에 버그가 있었던 입력 10개를 회귀 케이스로 보존
4. **비용 계산** — 샘플 기준 토큰 사용량 × 예상 MAU 호출 빈도 = 월 비용 추정

## 에러·엣지 케이스

- **JSON 파싱 실패**: 재시도 1회 → 실패 시 기본 응답(채점의 경우 "다시 한 번 답해볼까요?")
- **Rate Limit (429)**: 지수 백오프 3회 → OpenAI GPT-4o-mini 폴백
- **출력이 금지 주제(성인·혐오·정치 편향)에 닿음**: 필터로 차단, 기본 응답
- **한국어 품질 저하 조짐**: 샘플 세트로 재평가 후 프롬프트 rollback

## Sub-agent 위임 시

프롬프트 튜닝·대량 샘플 평가는 `general-task-execution`에 위임한다:
- 이 steering과 타깃 프롬프트 파일을 함께 전달
- 출력은 "개선안 + 근거 + 샘플별 Before/After" 형태로 요청
